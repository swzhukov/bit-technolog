# ADR-0011: v0.8 — полная переделка архитектуры (по разбору Сергея)

**Дата:** 2026-07-21
**Статус:** Принято (Sprint 1-8)
**Предпосылка:** Разбор v2 (концепция трёх слоёв) + 3 уточнения Сергея + 2 PDF реальных ТП

## Контекст

Сергей дал **разбор v2** (концепция v3, РС-фабрика, редизайн БД, ML/LLM лестница, ТЗ для разработчика). Затем **3 уточнения**:
1. РС-профили как **алгоритмические** (не LLM)
2. LLM модель выбирается **администратором** с отдельным правом
3. **Не обезличивать** данные

И **3 критичных замечания**:
- Модель + Шасси = один факт — одно место (редизайн БД)
- Исполнения (АЦ-8-40 разных внутренних номеров)
- Кооперация (давальческие схемы)
- Many-to-many деталей в сборках

Плюс **2 PDF** реальных ТП Техинкома (Упор продольный ЛМША.301314.010 + Растяжка пружинная ЛМША.301712.000) — это **эталоны** для RAG и few-shot.

Сергей сказал **"можешь сломать к чертям собачьим вообще всё что есть"** — карт-бланш на полную переделку.

## Решение

**Снести** v0.7 (4904 строк app.py, плоский FastAPI, денормализованная БД, mock_llm, статичный UI с моками). **Создать** v0.8 с чистой архитектурой по разбору v2.

### Принципы

**П1. Один факт — одно место.** Шасси — справочник + ссылка. Материал — справочник + ссылка. Модель изделия — отдельная сущность.

**П2. ref_1c у каждой сущности с двойником в 1С.** В препилоте — фиктивный UUID, но поле есть с первого дня.

**П3. Universal `items`** вместо `details`/`assemblies`/`purchased` (3 таблицы). Иерархия через `parent_id` (recursive). `level` поле для типа (detail/assembly/product). Справочники отдельно (chassis, materials, equipment, professions).

**П4. `bom_links` MANY-TO-MANY.** Деталь может входить в несколько сборок. `configuration_id NULL` = базовый состав модели; `configuration_id` = отличие исполнения.

**П5. Extension-атрибуты** — таблица `ext_attributes` (entity_type, ref_1c, attr_code, value). НЕ колонки в `items` для предметных параметров.

**П6. LLMProvider интерфейс** с назначением на задачу. Отдельное право `llm_admin`.

**П7. OneCGateway** — интерфейс с двумя реализациями. Бизнес-логика не знает, файл под ней или HTTP.

**П8. Детерминированная РС-фабрика** — никакого LLM в слое 2. Алгоритм по 8 осям профиля + отчёт трансформации.

**П9. ML/LLM лестница** (7 ступеней, в пилоте 1-4):
1. OCR+LLM-парсинг архивов
2. RAG + структурные фильтры
3. Динамический few-shot по эталонам
4. Петля обратной связи (edits → etalons)
5-7. Правила, регрессия, файн-тюнинг — после 3+ клиентов

**П10. Кооперация в РС** (по Сергею): `items.sourcing` (`make`/`buy`/`coop_da`/`coop_full`) + в ТК — операция "Передача в кооперацию" с полями контрагент/что-передаём/что-получаем. В РС — этап особого вида (по профилю).

### Альтернативы (отвергли)

- **Не ломать** — Сергей явно сказал "сломай всё", продолжение v0.7 приведёт к нагромождению
- **Переписать с нуля без incremental** — рискованно, нет защиты от регрессий
- **Удалить app.py целиком, написать новый** — сделаю, но в отдельной ветке, а app.py перепишу поэтапно
- **PostgreSQL** — после пилота (ADR-0011 принимает SQLite для препилота)
- **Файн-тюнинг** — после 3+ клиентов (ADR-0011 фиксирует)
- **OneCGateway HttpGateway реальный** — заглушка, реальная интеграция в пилот/продукт
