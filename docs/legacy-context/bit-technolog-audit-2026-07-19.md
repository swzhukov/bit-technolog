**Тесты:** 58/58 passing
**LLM:** YandexGPT (готовность), 1С-интеграция — нет
**Контекст:** 4 Sprint за 1 неделю (Sprint 0 → 5), demo-ready к пилоту

**Рекомендация:** добавить в промт явное `if uncertain, return null in field X`, и в RAG endpoint возвращать `{"warning": "мало данных для рекомендации"}` если `documents < 5`.

## 2. Правильность выбранных концепций, фреймворков, механизмов

### 2.1 Архитектурные решения

| Решение | Оценка | Обоснование |
|---------|--------|-------------|
| **FastAPI** (вместо Flask/Django) | ✅ Правильно | async, type hints, автодокументация, идеально для API-first + HTMX |
| **Jinja2 + HTMX** (вместо React/Vue) | ✅ Правильно | 240 строк CSS, минимум JS, работает без билда. Целевая аудитория не нуждается в SPA |
| **SQLite** (вместо PostgreSQL) | ⚠️ Для прототипа — да | Но pilot на 3-5 пользователей × 30 дней уже покажет concurrent write issues. Переезд на Postgres — 2 дня |
| **TF-IDF + cosine** (вместо OpenAI embeddings) | ✅ Правильно | On-prem, 0₽, для 50-1000 техкарт — accuracy сопоставима. Sprint 5 enterprise: апгрейд на embeddings |
| **OpenAI-совместимый SDK** (вместо anthropic SDK) | ✅ Правильно | Один клиент работает с YandexGPT, DeepSeek, OpenAI, 1BitAI. Замена LLM = 0 строк кода |
| **SQLite hybrid scoring** (material + equipment + mass) | ✅ Креативно | Не чистый TF-IDF, учитывает домен. Нужны веса по факту использования (сейчас 0.5/0.2/0.2/0.1) |
| **Автоиндексация на approve** (не на каждое generate) | ✅ Правильно | В индекс попадают только утверждённые человеком техкарты. Если 50% правятся — в индексе шум |
| **Inline-edit forms** (вместо prompt() JS) | ✅ Правильно | По правилу memory, работает в WebView Техинкома (если есть корп.браузер) |

### 2.2 Концептуальные решения

| Концепция | Оценка | Комментарий |
|-----------|--------|-------------|
| **3-step flow: analyze → draft → refine** | ✅ Правильно | Blueprint.io паттерн, снижает «шок от пустоты» у LLM, даёт пользователю agency |
| **2-3 alternative routes** | ⚠️ Спорно | Полезно для learning-by-comparison, но 99% пользователей выберут первый. **Лучше: 1 main + 1 «if regulatory» flag** |
| **Process-based pricing по цехам** | ✅ Правильно | CADDi pattern, критично для военной приёмки (калькуляция себестоимости по факту, не с потолка) |
| **Role model: draft → review → approved_chief** | ⚠️ Слабо | Нет реальной авторизации. Поле `chief="Баранов"` хардкодится в UI. **Нужен middleware с auth** |
| **30% acceptance rate** | ✅ Правильно | GitHub Copilot, NIO 33% — реалистичный benchmark. НЕ 80% (такого не бывает в engineering AI) |
| **Sprint 4 (1С) отложен** | ✅ Правильно | Для пилота 1С не нужен. Экспорт в Excel → ручной импорт в 1С — компромисс |
| **RAG + Few-shot** в промте | ✅ Правильно | ARKNESS paper показывает +25pp accuracy. Здесь TF-IDF как cheap analog |

### 2.3 Спорные / потенциально неправильные

#### ❌ 2.3.1 `MOCK_DETAILS` в 12 местах кода
`app.py:858, 925, 990, ...` — везде `next((d for d in MOCK_DETAILS if d["id"] == detail_id), None)`. Но `get_detail()` (374) уже делает SQL-запрос. **Дублирование** — `MOCK_DETAILS` используется для проверки существования, а реальные данные из `details` таблицы. Если данные в БД и `MOCK_DETAILS` разойдутся → баг.

**Fix:** везде использовать `get_detail()`, удалить `MOCK_DETAILS` импорт в runtime-эндпоинтах (оставить только в `generate_mock_draft()` для demo).

#### ⚠️ 2.3.2 `RAG` не валидирует качество индекса
`rag.rebuild_from_db()` возвращает количество, но **не проверяет**, что TF-IDF матрица не вырожденная. Если все тексты одинаковые (например, все «Сварка Кедр-300») — косинусное сходство всегда ~1.0, RAG бесполезен.

**Fix:** добавить в `rebuild_from_db()`:
```python
if len(drafts) < 5:
    log.warning("RAG: < 5 документов, similarity ненадёжна")
vocab_size = len(vectorizer.vocabulary_)
if vocab_size < 50:
    log.warning("RAG: словарь < 50 терминов, расширьте few_shot")
```

#### ⚠️ 2.3.3 `RAG hybrid scoring` веса захардкожены
`rag.py`: `0.5*sim + 0.2*material + 0.2*equipment + 0.1*mass`. **Без обоснования.** Если у техинкома 70% успеха зависит от оборудования, а не материала — веса неправильные.

**Fix:** сделать веса параметром `RAG_WEIGHTS` в `.env`, после пилота — A/B test.

#### ❌ 2.3.4 Нет fallback для LLM-парсинга
`app.py:1073` ловит `json.JSONDecodeError` → 500. Если модель вернёт `"```json\n{...}\n```"` с лишним пробелом — парсинг провалится. На YandexGPT в среднем 5-10% ответов оборачиваются в markdown.

**Fix:** добавить helper `parse_llm_json(text)`, который пробует: raw → strip ```json → strip ``` → first { to last }.

#### ❌ 2.3.5 `_get_param()` читает body **дважды** в случае ошибки JSON
`app.py:780-790` — после `except Exception: pass` пытается form-encoded. Но `body` уже прочитан. Если form-encoded и JSON парсятся в одном запросе (теоретически возможно в SPA) — будет неоднозначность. На практике не критично, но **логирование body на каждый запрос** (`log.info(f"{log_name}: body={body!r}")`) — **утечка PII** в проде (могут быть ФИО конструкторов, описание деталей).

**Fix:** убрать логирование body в проде (`if log_name and not DEMO_MODE: log.debug(...)`), оставить только `log_name` + факт успеха.

## 3. Реализация

### 3.1 Критичные баги (блокируют прод)

#### 🔴 C1: Нет аутентификации
`/detail/detail-001` — открыт всем. В `approve-chief` хардкод `chief="Баранов"`. Любой может утвердить что угодно.

**Риск для пилота:** Главный технолог увидит «не он» утвердил → потеря доверия.

**Fix (минимум):** HTTP Basic Auth с переменной `PILOT_USERS=user:pass,admin:pass` в `.env`. 30 строк кода.

**Fix (правильно):** middleware с JWT + ролями (technologist / chief / admin). 1-2 дня.

#### 🔴 C2: `_get_param` не валидирует тип `detail_id`
`/api/generate?detail_id=<script>` → уйдёт в `next((d for d in MOCK_DETAILS if d["id"] == "<script>"))` → None → 404. **OK** для GET. Но `/api/apply-similar` записывает `source_id` в БД без экранирования → потенциальная XSS в `/audit` (если source_id отрисовывается без эscape).

**Fix:** В Jinja2 уже автоэскейпится. Но в `add_history()` `details=json.dumps({"source_id": source_id})` — если кто-то сделает GET `/audit` и source_id окажется с `<script>` — JSON валидный, render безопасен. **OK, не баг, но ловушка на будущее.**

#### 🔴 C3: `get_detail()` создаёт **новое соединение** для `PRAGMA`
`app.py:381-382`:
```python
cols = [d[1] for d in sqlite3.connect(DB_PATH).execute("PRAGMA table_info(details)").fetchall()]
```
**Открывает SQLite соединение, делает запрос, забывает закрыть** (нет `conn.close()`). Утечка файловых дескрипторов. На пилоте за день — десятки лишних дескрипторов. На проде через неделю — `Too many open files`.

**Fix:** вынести в helper:
```python
def get_table_columns(table: str) -> list:
    conn = get_conn()
    cols = [d[1] for d in conn.execute(f"PRAGMA table_info({table})").fetchall()]
    conn.close()
    return cols
```

#### 🔴 C4: `openai` клиент создаётся **на каждый вызов**
`app.py:882, 942, 1017, ...` — `client = OpenAI(base_url=..., api_key=..., timeout=...)` создаётся 4+ раз. **Это 200ms × N вызовов.** На пилоте с 50 деталями — лишние 10 секунд.

**Fix:** глобальный singleton `LLM_CLIENT` через `functools.lru_cache` или модуль-переменную.

#### 🟠 C5: SQLite без `journal_mode=WAL`
`init_db()` не выставляет WAL. При concurrent writes (3 технолога одновременно) — `database is locked`. На пилоте с 3-5 пользователями **точно случится**.

**Fix:** добавить в `init_db()` сразу после `CREATE TABLE`:
```python
conn.execute("PRAGMA journal_mode=WAL;")
conn.execute("PRAGMA synchronous=NORMAL;")
```

#### 🟠 C6: `LLM_DAILY_LIMIT_RUB` не обнуляется в полночь
`get_daily_cost()` суммирует `cost_rub FROM llm_calls WHERE created_at > date('now', 'start of day')`. Если 19:00 потратили 180₽, оставили на ночь, утром — стартуем с 180₽ → лимит 200₽ → 20₽ остаётся.

**Fix:** OK, это by design (откат за вчера). Но добавьте `LIMIT_HARD_STOP=true` для пилота.

### 3.2 Существенные баги (не блокируют, но неприятно)

#### 🟡 M1: `audit.html` рендерит `{{ e.details }}` как строку
`templates/audit.html:23` — `<code>{{ e.details or '{}' }}</code>`. Но `e.details` это JSON-строка вида `{"reason": "текст с кавычками"}`. HTML-рендер: `{"reason": "текст с кавычками"}` — выглядит как мусор.

**Fix:** парсить в шаблоне и рендерить как `<pre>{{ e.details | fromjson | tojson(indent=2) }}</pre>`. Или просто `<pre>{{ e.details }}</pre>` (multi-line).

#### 🟡 M2: `add_history` принимает `details: dict`, но **JSON дамп с ASCII-квотами**
`app.py:747` — `details = json.dumps(details, ensure_ascii=False) if details else ""`. OK. Но `reason[:500]` в `/api/feedback:1098` — обрезаем до 500 символов **без учёта UTF-8**. Можно порезать посреди кириллического символа.

**Fix:** `reason.encode('utf-8')[:500].decode('utf-8', errors='ignore')`.

#### 🟡 M3: `apply-similar` **не делает RAG auto-index**
`app.py:1448` — после успешного `apply-similar` нет вызова `rag_index_detail()`. Значит, применённая техкарта не попадёт в RAG до следующего `approve` или rebuild.

**Fix:** добавить `try: rag_index_detail(detail_id) except: pass` в конец `apply_similar`.

#### 🟡 M4: 3-step flow кнопки передают `answers: "{}"` хардкодом
`detail.html:79, 82` — `hx-vals='{"detail_id": "...", "answers": "{}"}'`. Это **не работает**: после шага 1 ответы появляются в `questions-area`, но **не передаются** в шаг 2/3. Юзер кликает «Refine» → LLM получает пустые `answers` → refine как без анализа.

**Fix:** нужен `localStorage` или hidden input, обновляемый из JS после шага 1. Либо POST шага 1 сохраняет answers в БД, шаг 2/3 читает.

#### 🟡 M5: `batch-generate` **не логирует в `llm_calls`**
`app.py:1407` — при DEMO_MODE пишет `save_draft()` но не `log_llm_call()`. Pilot dashboard `total_successful_gens` покажет 0.

**Fix:** добавить `log_llm_call(did, "demo", "", "", llm_output_json, True, 0, 0, 100, 0.0, None)` в DEMO ветку.

#### 🟡 M6: `RAG` падает при **пустой БД**
`rag.py:rebuild_from_db()` — если 0 drafts, `vectorizer.fit_transform([])` упадёт. Тест `test_rag_rebuild_builds_index` проходит только потому что в БД уже есть 4 мок-детали. На свежей БД — 500.

**Fix:**
```python
if len(drafts) == 0:
    self.loaded = False
    return 0
```

#### 🟡 M7: `equipment.json` и `structure.json` **не валидируются при старте**
`app.py:78-83` — `json.load(f)` без `try/except`. Если файл повреждён — `JSONDecodeError` при импорте → app не стартует. **OK** для прототипа, но в проде нужна graceful degradation.

#### 🟡 M8: `rag.py:hybrid_score` принимает **None** для массы
Если `mass_kg` нет в `details` (например, новая деталь без указания массы) — `cand_mass = None`, сравнение `abs(query_mass - cand_mass) > 50` упадёт с TypeError.

**Fix:** `if query_mass is None or cand_mass is None: mass_score = 0.5; else: ...`.

### 3.5 Перформанс

```python
details = get_all_details()
if details:
    ids = tuple(d["id"] for d in details)
    placeholders = ",".join("?"*len(ids))
    rows = conn.execute(f"SELECT detail_id, status FROM drafts WHERE detail_id IN ({placeholders})", ids).fetchall()
    status_map = {r[0]: r[1] for r in rows}
    for d in details:
        d["status"] = status_map.get(d["id"], "new")
```

## 4. UX/юзабилити для машиностроения

### 4.1 Контекст пользователя

**Целевой технолог Техинкома:**
- 45-60 лет, 20-30 лет стажа
- Работает в КОМПАС-3D / 1С:ERP / Excel — НЕ в браузере
- Очки, плохое зрение (близорукость/дальнозоркость)
- Нетерпелив, ценит «короткий путь»
- Не любит «детский» дизайн (эмодзи, яркие цвета)
- Главный страх: «AI подставит, а отвечать мне»

**Текущий UI не учитывает:**
- ❌ Шрифт 14px (нужен 16-18px для 50+)
- ❌ Эмодзи 🔴🟡🟢 в badge (нужны текстовые метки «Новый/Черновик/Утверждён»)
- ❌ Цветовая кодировка только (для дальтоников — бесполезна)
- ❌ Нет keyboard shortcuts (Ctrl+Enter = approve, Ctrl+S = save, Esc = cancel)
- ❌ Модальные окна (нет, но `hx-confirm` — не адаптировано)
- ❌ Нет «печатной версии» (для подписи в бумажный журнал)

### 4.2 Критичные UX-проблемы

#### 🔴 U1: На `detail.html` 5+ раскрывающихся секций с одинаковым summary
<details> с summary: «Правила технолога», «Экономика», «Sprint 1: 3-step generation», «Sprint 2: Похожие техкарты», «Sprint 3: Альтернативы». **Технолог не знает что внутри, не открывает.**

**Fix:** заменить на **tab-bar в стиле 1С** (вертикальный sidebar):
```
[📊 Сводка] [⚙ Операции] [🛤 Маршрут] [💰 Экономика] [📜 Версии] [🤖 AI-помощь]
```
AI-помощь = 1 вкладка со всем (analyze, draft, alternatives, RAG).

#### 🔴 U2: Sprint 1/2/3 в summary — **выдаёт self-reference**
Технолог видит «Sprint 1: 3-step generation» и думает «что за спринт?». Это **внутренний термин разработчика**.

**Fix:** переименовать:
- «Sprint 1: 3-step generation» → «🤖 AI-помощник: уточнить → draft → полная ТК»
- «Sprint 2: Похожие техкарты (RAG)» → «🔍 Похожие из истории»
- «Sprint 3: Альтернативные маршруты» → «🔀 Другие варианты маршрута»

#### 🔴 U3: 3-step flow кнопки «⚡ Draft» «✨ Refine» — **непонятный workflow**
Технолог кликает «⚡ Draft» → получает 3 операции → и что? «✨ Refine» — это **продолжение или с нуля**?

**Fix:** визуальный progress indicator (1 → 2 → 3) + кнопка «Шаг 3 доступен после ответа на Шаг 1». **Или скрыть Шаг 3, пока Шаг 1 не отвечен.**

#### 🟠 U4: `/index` — все 575 деталей в одной таблице
При 575 строках — скролл на 3 экрана. Технолог устаёт.

**Fix:** пагинация (50 на страницу) + фильтры (статус, материал, шасси).

#### 🟠 U5: `draft.summary.total_hours` без размерности
`<div class="metric-value">{{ draft.summary.total_hours }}</div>` — 2.5 чего? Часа? Дней?

**Fix:** `<div class="metric-value">{{ total_hours }} <small>ч</small></div>` (есть в CSS `.metric-value` но не используется). Или явно подпись «часов общее».

#### 🟠 U6: Operations edit — **inline form с 4 input**
`detail.html:270-298` — для каждой операции 3 inline-edit формы (name, equipment, duration). **Клик → input → кнопка ✏️ → submit.** Технолог скажет «да проще Excel».

**Fix:** один клик по полю → сразу `<input>` (без кнопки), Enter = save, Esc = cancel. Стандартный паттерн Trello/Notion.

#### 🟡 U7: Warnings показаны как «warning-card» с цитатой
Хорошо для design portfolio, плохо для работы. Технолог хочет **action**: «нажать → исправить → перегенерировать».

**Fix:** в `warning-card` добавить кнопку «✏️ Исправить» которая открывает inline-form на проблемном поле.

#### 🟡 U8: Цвет уверенности 🟢🟡🔴 — не достаточно
`{% if op.confidence >= 80 %}🟢 ... {% elif 60 %}🟡 ... {% else %}🔴` — OK для UI, но **не показывает почему**. Технолог: «почему 65%? Откуда?»

**Fix:** tooltip или expandable: «Низкая уверенность: время оценено по аналогу, требует уточнения ТО».

#### 🟡 U9: Нет «массовых действий» в `/index`
Если 50 деталей в статусе «new» — технолог хочет «сгенерировать все», «отправить все на проверку», «утвердить выбранные». Сейчас — только по одной.

**Fix:** `batch-generate` уже есть. Добавить checkbox в таблицу + toolbar «С отмеченными: …».
