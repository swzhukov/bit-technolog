### Фаза 2. MVP v0.1 (3-4 недели) 🟢 МОЖНО СТАРТОВАТЬ

**Стек:** Python 3.11+, FastAPI, SQLite, HTMX, Docker Compose. On-premise.

**Компоненты:**

1. **Watcher КОМПАС-3D** (3-5 дней)
   - Следит за watch-папкой (.m3d/.cdw)
   - Парсит свойства: наименование, обозначение, масса, материал, габариты, ГОСТ
   - Отправляет событие в очередь

2. **LLM-агент с промтом v0.2** (3-5 дней)
   - Включает **Few-shot** из техкарты 4c85941a (Кедр-300, операции 015-040)
   - Включает **RAG-поиск** аналогов (по ведомости 89316e6a)
   - Возвращает JSON по 6 вкладкам
   - Latency: ≤ 30 сек

3. **RAG-индексация** (2-3 дня)
   - ChromaDB + OpenAI/Anthropic embeddings
   - Чанки: ведомость 89316e6a (575 ДСЕ) + наряд 2d876715 + 2 техкарты
   - Поиск по обозначению, материалу, габаритам

4. **Справочник оборудования** (1-2 дня)
   - Кедр-300 (сварочный аппарат) + 5-10 других
   - Формат: название, группа, операции, габариты, привязка к участку

5. **Web UI** (5-7 дней)
   - По UX-флоу v6: 12 экранов
   - HTMX для интерактивности (без React)
   - Список деталей, карточка, 6 вкладок, действия

6. **Экспорт в Excel** (1-2 дня)
   - По шаблону 1С:ERP (формат ГОСТ 3.1105-2011, если возможно)
   - Или простой Excel с операциями

7. **Тест на 3-5 деталях** (2-3 дня)

## 6. РИСКИ И FALLBACK

### Риск 1: Заболотный не согласует проект
**Fallback:** Сценарий D из v5 — продаём как услугу внутри ДКП-5, без отдельного продукта.

### Риск 2: Не получим .m3d/.cdw
**Fallback:** Начинаем с **PDF-сканов** (через OCR, как у ИИ-Технолога). Менее точно, но рабоче.

### Риск 3: Точность MVP < 50%
**Fallback:** Итерируем промт 2-3 недели. Если не помогает — переключаемся на Few-shot с техкартами Техинкома (их уже 2, нужны ещё 1-3).

### Риск 4: Баранов не будет пользоваться
**Fallback:** Custdev с технологом. Ищем реальную боль. Возможно, нужен другой пользователь.

### Риск 5: Конкурент (ИИ-Технолог) уже у Техинкома
**Fallback:** Проверить через прямой вопрос. Если да — другая стратегия (дополнять, а не внедрять с нуля).
