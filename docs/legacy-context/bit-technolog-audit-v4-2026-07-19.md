## 3. Найдено в этом цикле (v4)

### 🔴 V1: `accepted_target: 50%` в коде, но в Product Design v0.4 — **30%**

`app.py:782` — `"accepted_target": 50` и UI на `pilot.html:31` — `> 50%`. **Но:**
- В Product Design v0.4 §3.3 KPI: «30% acceptance rate (GitHub Copilot benchmark)»
- В моих memory записи: «30% — industry standard»
- **Код говорит 50%, документ говорит 30%**

Это **влияет на пилотное решение** — если Баранов увидит «20% принято» и target 50% — пилот «провалился», хотя по индустриальному стандарту 20% уже хорошо.

**[СЕРГЕЮ-9]**: `accepted_target: 30%` или `50%`? Меняю код на 30% — это индустриальный benchmark (GitHub Copilot, NIO 33%). Если у Техинкома внутреннее требование 50% — оставлю 50% + Сергей переутвердит Product Design.

### 🟠 V2: `pilot.html` target "цель < 60 мин" но current 4-8ч = 240-480 мин. 60 мин — оптимистично
`pilot.html:11` — `> 60 мин (текущее 4-8ч)`. 4-8ч = 240-480 мин. Цель 60 мин = 4-8x ускорение. В Product Design v0.4 §3.3: «time-to-techcard 4-8h → 30 min (8-16x faster)».

**[СЕРГЕЮ-10]**: target 30 мин или 60 мин? Реалистичнее 60 мин (включает review).

### 🟡 V3: 30+ вхождений `JSONResponse({"error": "..."})` — pattern без status code context
Когда на пилоте упадёт `500 Internal Server Error` — в логах будет только `{"error": "..."}` без request_id, без user_id. Не критично, но debugging будет мучительным.

### ⚪ V4: pilot.html не имеет графиков (только цифры)
Для презентации Баранову — лучше Chart.js график «% принятых по неделям». Но **после пилота** когда будет 4+ недели данных.

### ⚪ V5: `/llm-debug` показывает последние 50 вызовов — без фильтра по детали, без экспорта
Удобно для разработчика, не для технолога. OK.
