# БИТ.Технолог — Архитектура обучения

> **Дата:** 2026-07-17
> **Задача:** спроектировать, как AI должен "учиться" на техкартах Техинкома, чтобы не лепить "упор продольный" на любую деталь.
> **Сейчас:** few-shot в промте (2 техкарты) + таблица `edits` (правки) — НЕ работает в масштабе.

---

## 1. TL;DR — что делаем

**Не RAG в чистом виде. Не fine-tuning. Гибрид: Memory-Augmented Generation (MAG).**

```
Суть: AI получает контекст из ТРЁХ источников одновременно:
  1. Few-shot в промте (2 эталонные техкарты) — базовое "формат"
  2. RAG-примеры (top-3-5 похожих техкарт) — "как мы делали похожее"
  3. Извлечённые правила (из правок технолога) — "что точно НЕ делать"
```

**Почему не RAG сам по себе:**
- 575 деталей в ведомости — но только 50-100 имеют **полные** техкарты (остальное только нормы времени)
- RAG на тексте без структуры (JSON) даёт "похожие описания", не "похожие операции"
- LLM и так галлюцинирует — RAG без фильтрации усиливает мусор

**Почему не fine-tuning:**
- Нужно 1000+ размеченных примеров (у нас 50-100)
- Каждое изменение в политике → переобучение за $$ 
- Нет возможности A/B тестировать варианты

**Что это даёт:**
- Через 2 недели пилотa: 60-70% принятых AI-операций (вместо 30-40% сейчас)
- Через 100 одобренных техкарт: 75-85% принятых
- Через 500: 85-90% (и тогда AI = полноценный junior-технолог)

### 2.2. Какой пример когда подавать

| Ситуация | Что подать |
|---|---|
| Кронштейн для АЦ-6-40 | Слой 1 + top-3 кронштейна из ведомости + правило "Кронштейн замка = 4.5ч" |
| Плита новая | Слой 1 + top-3 плиты + правила (если есть про плиты) |

| Неизвестная деталь | Только слой 1 (few-shot) — AI должен сам спросить |

```
            └─────────────────────┘
```

### 3.1. Подсистемы

**A. Indexer (offline)**
- Берёт утверждённые черновики
- Извлекает embedding детали (designation + material + type)
- Сохраняет в vector store: {detail_id, embedding, material, equipment, ops_count, hours}
- Триггер: при `approve` или `approve-chief`

**B. Retriever (online, при generate)**
- Embedding входной детали
- Top-K=5 кандидатов
- Re-rank: similarity × material_match × equipment_match × operations_pattern
- Возвращает top-3 (чтобы не перегружать промт)

**C. Rule Extractor (online, при edit)**
- При правке: field, old, new
- Группирует по field: "field X всегда меняется с A на B"
- При >= 3 повторений → правило с confidence=0.7
- При >= 5 → confidence=0.9
- Хранится в `rules` table

**D. Rule Applier (online, при generate)**
- Перед запросом: проверка каждой AI-операции против rules
- Если нарушает rule с confidence > 0.7 → warning
- Если > 0.9 → блокирует (требует override от технолога)

## 4. ТЕХНОЛОГИЧЕСКИЙ СТЕК

### 4.1. Embeddings (что переводит текст в числа)

| Вариант | Качество русского | Стоимость | Локальность |
|---|---|---|---|
| **OpenAI text-embedding-3-small** | 🟢 Хорошее | 🟢 $0.02/1M токенов (~575 деталей = $0.05 разово) | ☁️ API |
| YandexGPT embeddings | 🟡 Нативный русский | ❓ Нет в открытом доступе | ☁️ |
| bge-m3 (локально) | 🟡 Среднее на русском | 🟢 Бесплатно | 🏠 On-premise |
| E5-large-v2 | 🟡 | 🟢 Бесплатно | 🏠 |

**Решение: OpenAI text-embedding-3-small для старта.**
- $0.05 разово индексировать 575 деталей
- ~$0.001 за каждый retrieve
- Качество для русского языка приемлемое (мы тестировали)
- Если Техинком скажет "только on-premise" — переключимся на bge-m3

**Embedding-что:**
- Текст: `"{designation} {name} {material} {model} {operations_summary}"`
- НЕ embed операций (JSON) — embeddings не понимают структуру
- Embed тип детали: "Кронштейн для АЦ-6-40" → vector(1536)

### 4.2. Vector Store (где хранить embeddings)

| Вариант | Плюсы | Минусы |
|---|---|---|
| **ChromaDB** | 🟢 Embeddings, легко, SQLite-style | 🟡 Молодой |
| Qdrant | 🟢 Production-ready, фильтры | 🟡 Сложнее setup |
| pgvector (Postgres) | 🟢 Надёжно | 🟡 Нужен Postgres (сейчас SQLite) |
| Pinecone | 🟢 SaaS, не надо setup | 🔴 Стоит $ + on-premise нельзя |
| FAISS (Meta) | 🟢 Быстро | 🔴 Только vectors, без метаданных |

**Решение: ChromaDB для прототипа.**
- Pip install, embed прямо в коде
- Persistent (файл на диске, не in-memory)
- ~5МБ для 575 деталей
- Metadata filtering: `where={"material": "Сталь 09Г2С"}`
- Когда переедем на Postgres — заменим на pgvector

### 4.3. Hybrid Search (как искать)

**Не только embedding — комбинируем 4 сигнала:**

```python
def score(example, query):
    sim = cosine_similarity(embed(example), embed(query))  # 0-1
    mat = 1.0 if example.material == query.material else 0.5
    eq = jaccard(example.equipment, query.equipment)  # 0-1
    op = 1.0 if example.op_count_range == query.op_count_range else 0.7
    return 0.5 * sim + 0.2 * mat + 0.2 * eq + 0.1 * op
```

Top-3 скора → в промт как пример.

### 4.4. Где запускаем

| Компонент | Где |
|---|---|
| ChromaDB | Тот же сервер, в файловой системе |
| Embeddings API | OpenAI API (запросы наружу, 1-2 сек) |
| Rules DB | Та же SQLite |
| LLM | YandexGPT (on-premise опционально) |

## 5. МЕТОДОЛОГИЯ ОБУЧЕНИЯ

### 5.1. Триггеры обновления (когда добавляем/обновляем)

| Событие | Действие |
|---|---|
| `approve` (технолог) | Index: добавить embedding в vector store, status="approved" |
| `approve-chief` (начальник) | Re-index с повышенным весом (chief-validated = 1.5x) |
| `edit` (правка) | Если field+old+new уже встречалось 2+ раза → создать rule |
| `submit-for-review` | Не обновляет (только отметка) |
| Раз в неделю | Re-cluster, удаление дублей (cosine > 0.95) |

### 5.2. Качество правил (как не замусорить)

Правило создаётся ТОЛЬКО если:
- field встречалось в правках 3+ раз
- old_value и new_value консистентны (всегда одно old → одно new)
- reason (если есть) — содержательный
- Confidence = min(1.0, 0.5 + 0.1 * count)

Правило **архивируется** (не удаляется), если:
- 30 дней не подтверждалось
- Есть override от технолога (значит правило устарело)

### 5.3. Anti-rules (защита от мусора)

Если технолог переопределил правило, мы НЕ стираем его сразу, а:
- Помечаем `disputed=true`
- Confidence снижается на 0.1 за каждый override
- Если confidence < 0.3 → помещаем в "archive"

Это важно, чтобы AI не "разучился" правилу из-за одного outlier.

### 5.4. Тестирование (как проверять что обучение работает)

**Сценарии A/B:**
- 10 деталей → генерируем 2 черновика (с RAG и без)
- Технолог не знает какой какой
- Считаем: какой вариант меньше правок → это и есть "обученная" версия
- Через 30 деталей — статистически значимый вывод

**Метрика успеха:**
- `accepted_ops_pct` (цель: рост неделя к неделе)
- `edits_per_draft` (цель: падение)
- `time_to_card_min` (цель: падение)

Если метрики не улучшаются 4 недели подряд — обучение не работает, нужно пересмотреть стратегию.

## 8. ЧТО НЕ ДЕЛАЕМ (и почему)

| Подход | Почему нет |
|---|---|
| **Fine-tuning модели** | Нужно 1000+ примеров. У нас 50-100. Стоимость: $1000-$10000 за обучение + 1-2 недели. Не оправдано. |
| **RLHF** | Сложно реализовать, нужен штат ML-инженера. YandexGPT не даёт дообучать. |
| **Embeddings для операций (JSON)** | Embeddings не понимают структуру. Операция = список ключей, embedding схлопывает информацию. Лучше: hybrid (embedding детали + структурный матч операций). |
| **Fine-tuning embeddings на русском** | Нет готовых моделей. bge-m3 — лучший open source, но не "наш". |
| **Graph-based retrieval (knowledge graph)** | Overkill для 575 деталей. ChromaDB + hybrid scoring хватит. |
| **Multi-modal embeddings (текст + изображение чертежа)** | Нужен Watcher КОМПАС-3D сначала. Сейчас нет. |
