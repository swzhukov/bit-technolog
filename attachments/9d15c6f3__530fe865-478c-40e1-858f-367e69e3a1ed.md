## 1. ЦЕЛЬ ПРОДУКТА

**AI-помощник технолога для ускорения создания техкарт** в 1С:ERP для мелкосерийного машиностроения с индивидуальными заказами.

**Пилотный клиент:** Техинком (ООО «ПК Техинком»).

**Не «AI-надстройка для КИВТ».** Не «AI-ассистент внедрения 1С:ERP». А именно **помощник технолога**.

## 7. РЕАЛЬНАЯ БОЛЬ ТЕХИНКОМА (уточнённая)

### Из транскрибации + оглавления проекта:

**Боль №1: Техкарты для КиВТ — формальная кипа документов**
- Техкарты есть, но формальные — отдельная кипа, не связанная с реальным производством
- УМК — нормальные (там 2 дня, всё просто)
- АЦ-8-40 — нормальные (потому что часто делают)
- Новые модели КиВТ — каждый раз заново, долго, шаблонно

**Боль №2: Новые модели КиВТ делаются впервые**
- Каждый заказ индивидуальный
- Технолог составляет техкарту **посмертно** (после того, как изделие уже в производстве)
- Реальные нормы не считаются и не используются
- Извещения об изменениях — устно

**Боль №3: Конструктор и технолог работают в параллельных вселенных**
- Конструктор рисует в КОМПАС-3D (5 человек)
- Технолог составляет техкарту (3 человека)
- Связь слабая
- Изменения в КД не доходят до технолога формально

**Боль №4: Контрольные точки не оцифрованы (для ГД)**
- ГД не имеет инструмента видеть статус каждой машины
- Контрольные точки (ОТК, покраска, испытания, электрика) не привязаны к РС

**Боль №5: Электрика — чёрный ящик**
- Жгуты/пульты не интегрированы в общий контур
- Непонятно, как контролировать и управлять

**Боль №6: Нормативная длительность по контрольным точкам не проставлена**
- Из презентации КИВТ: «Для каждого этапа указывается время в часах и привязка к рабочему центру»
- Сейчас этого нет

**Боль №7: Загрузка рабочих центров и нормирование — мёртвое дитя**
- Из саммари встречи: «Загрузку рабочих центров и углубление нормирования до детали — сейчас это «мёртвое дитя» для малой серии»
- **Не наша задача в MVP**

**Боль №8: Учёт трудозатрат для ГОЗ — на бумажках**
- Военная приёмка требует чётких человеко-часов на машину
- **Не наша задача в MVP**

## 8. АРХИТЕКТУРА MVP (исходная)

```
Конструктор (КОМПАС-3D)
  └─ Сохраняет .m3d/.cdw в watch-папку
       │
       ▼
Python Watcher (FastAPI)
  ├─ Парсит свойства детали (масса, материал, ГОСТ, габариты)
  └─ Передаёт в LLM-агент
       │
       ▼
LLM-агент (Claude Sonnet / YandexGPT Pro)
  ├─ Получает: свойства детали + АЦ-8-40 как образец + справочник оборудования + справочник материалов
  ├─ Генерирует: черновик техкарты с правильной структурой
  ├─ Считает: время операции (по нормативу / по аналогии / экспертная оценка)
  └─ Выбирает: оборудование из парка предприятия
       │
       ▼
Web-интерфейс (технолог)
  ├─ Видит: черновик + исходные данные + уверенность LLM
  ├─ Правит: операции, материалы, время
  └─ Утверждает → экспорт в 1С
       │
       ▼
1С:ERP 2.5 (с доработками Первого Бита)
  └─ Документ "Технологическая карта" + РС
```

## 9. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ MVP

### Блок 1. Сбор данных из КОМПАСа (приоритет: Высокий)
- Watcher следит за 3-5 папками (по конструкторам)
- Парсит свойства детали: наименование, обозначение, масса, материал, габариты, ГОСТ
- Отправляет событие «Новая деталь» в очередь

### Блок 2. Генерация черновика техкарты LLM (приоритет: Высокий)
- На входе: свойства детали + АЦ-8-40 как образец + справочник оборудования + справочник материалов + шаблон типовой техкарты
- LLM формирует структурированный черновик:
  - Список операций (раскрой, токарная, фрезерная, сварка, покраска, сборка, электрика)
  - Оборудование для каждой операции
  - Время на операцию с источником («по нормативу» / «по аналогии с АЦ-8-40» / «экспертная оценка»)
  - Материалы (заготовка, вспомогательные)
  - Контрольные точки (ОТК, испытания)
- Для каждой оценки времени — **уверенность 0-100%** и **почему**

### Блок 3. Справочники / Knowledge Base (приоритет: Средний)
- Справочник оборудования Техинкома (название, тип, макс. размер заготовки, точность, типовые операции)
- Справочник материалов (название, плотность, цена, типовые заготовки)
- **АЦ-8-40 как образец** (полная техкарта с правильной структурой)
- База утверждённых техкарт прошлых заказов (для RAG)

### Блок 4. Интерфейс технолога (приоритет: Высокий)
- Список деталей в работе
- Карточка детали: свойства + черновик + история правок
- Кнопки: «Утвердить», «Отклонить», «Запросить доработку у конструктора»
- Комментарии

### Блок 5. Интеграция с 1С:ERP 2.5 (приоритет: Средний)
- На MVP: Excel по шаблону 1С → ручная загрузка (без 1С-разработчика)
- Потом: JSON → обработка загрузки в 1С с учётом реестра доработок на 08.04.2026

### Блок 6. Версионность и извещения (приоритет: Низкий на MVP)
- При изменении .m3d/.cdw — автоуведомление технологу «Деталь изменилась, техкарта требует пересмотра»
- История версий (последние 5)

## 10. НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

- Латентность генерации черновика: ≤ 30 секунд
- **On-premise, на сервере заказчика** (требование спецтехники / ГОЗ)
- Разграничение доступа: технолог видит всё, конструктор — только свои детали
- Аудит: кто, когда, что изменил
- Простой запуск: docker-compose на одном сервере
